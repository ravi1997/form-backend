## Registration

| Property | Value |
|----------|-------|
| Blueprint | `form_bp` (shared) |

---

## Overview

Provides LLM-backed summarization of form responses. Supports both synchronous (JSON response) and streaming (SSE) modes. Requires at least 2 responses to generate a summary. Enforces organization isolation via form lookup.

**Known issue:** `app_logger` and `error_logger` are referenced without explicit import in `summarization.py`. See `risks-and-gaps.md` R-07.

## Route Reference

### POST /form/api/v1/forms/`<form_id>`/summarize

**Summary:** Generate a summary of form responses using the LLM.

**Authentication:** `@jwt_required()`

**Request body:**
```json
{
  "response_ids": ["uuid1", "uuid2", "uuid3"]
}
```

`response_ids` is optional. If omitted, all responses for the form are summarized.

**Pre-conditions:**
1. `form_id` must be a valid UUID (400 if not)
2. Form must exist within the user's organization (404 if not)
3. Must have at least 2 responses (400 if fewer)

**Behavior:**
- Instantiates `SummarizationService()`
- Calls `summarizer.summarize_form(form_id, response_ids=response_ids)`
- The service uses the Ollama LLM to generate a text summary of response data

**Success response (200):**
```json
{
  "success": true,
  "summary": "The majority of respondents reported fever and cough as primary symptoms. Most patients are in the 30-50 age range...",
  "form_id": "uuid"
}
```

**Error responses:**
- `400` — Invalid UUID format: `{"error": "Invalid form ID format", "success": false}`
- `400` — Insufficient responses: `{"error": "At least 2 responses required for summarization", "success": false}`
- `404` — Form not found: `{"error": "Form not found", "success": false}`
- `500` — Summarization error (LLM unavailable, timeout, etc.): `{"error": "...", "success": false}`

**Note:** This route returns raw `jsonify()` rather than `success_response()`.

---

### POST /form/api/v1/forms/`<form_id>`/summarize-stream

**Summary:** Generate a summary with Server-Sent Events streaming response.

**Authentication:** `@jwt_required()`

**Request body:** Same as `/summarize`
```json
{ "response_ids": [] }
```

**Pre-conditions:** Same as `/summarize` — form existence and org isolation checked.

**Behavior:**
- Returns `Content-Type: text/event-stream` response
- Uses `stream_with_context(generate())` generator
- The generator calls `SummarizationService.summarize_form()` and yields the result as a single SSE event
- Current implementation generates the full summary first, then emits it as one event (not true token-by-token streaming)

**Response format (SSE):**
```
data: {"content": "The majority of respondents...", "done": true}

```

(Note the double newline at end of each SSE event)

**On error, yields:**
```
data: {"error": "LLM unavailable", "done": true}

```

**Client implementation notes:**
- Connect with `EventSource` (browser) or an SSE client library
- The current implementation emits one event and closes the stream
- Future versions may stream tokens as they are generated by Ollama

## AI Provider Notes

- Summarization uses the `default_model` configured in `OllamaService` (default: `llama3.2`)
- If Ollama is unavailable, summarization routes return 500
- Check AI health before relying on summarization: `GET /form/api/v1/ai/health`
- Summaries are generated on-demand; they are not cached at the route level
